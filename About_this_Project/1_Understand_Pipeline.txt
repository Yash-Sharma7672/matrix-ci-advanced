 # 🟡 PYTHON MATRIX JOB
  python-test:
    runs-on: ${{ matrix.os }}

    strategy:
      fail-fast: false
      matrix:
        os: [ubuntu-latest, windows-latest]
        python-version: [3.8, 3.9, 3.10]

    steps:
      - name: Checkout code
        uses: actions/checkout@v3

      # Setup Python
      - name: Setup Python
        uses: actions/setup-python@v4
        with:
          python-version: ${{ matrix.python-version }}

      # Cache pip dependencies
      - name: Cache pip
        uses: actions/cache@v3
        with:
          path: ~/.cache/pip
          key: ${{ runner.os }}-pip-${{ matrix.python-version }}

      - name: Install dependencies
        run: |
          cd python-app
          pip install -r requirements.txt

      - name: Lint code
        run: |
          cd python-app
          flake8 .

      - name: Run tests
        run: |
          cd python-app
          pytest

                                        Explanation
-------------------------------------------------------------------------------------------------------
🔶 python-test:

👉 This is the job ID
    Every workflow contains jobs
    This job is named python-test
Why it matters:
    Used internally by GitHub Actions
    Helps in logs, debugging, dependencies between jobs
-----------------------------------------------------------------------------------------------------

🔶 runs-on: ${{ matrix.os }}

👉 Defines the execution environment (runner)
        Instead of hardcoding OS → it uses matrix value

What happens:
If matrix has:
    os: [ubuntu-latest, windows-latest]
Then GitHub creates:
    1 job on Linux
    1 job on Windows

👉 Combined with Python versions → multiple jobs

-------------------------------------------------------------------------------------------------------
🔶 strategy:

👉 Controls how multiple jobs are created and executed

-------------------------------------------------------------------------------------------------------
🔸 fail-fast: false

👉 Default is true, but here it's disabled

Meaning:
    If one job fails → others still continue
Why used:
In matrix testing:
    You want full visibility
    Not just first failure

👉 Production benefit:
You get complete compatibility report, not partial

-------------------------------------------------------------------------------------------------------
🔸 matrix:

👉 This is the core scaling engine
It generates combinations of environments.

-------------------------------------------------------------------------------------------------------
🔹 os: [ubuntu-latest, windows-latest]

👉 Defines operating systems
    ubuntu-latest → Linux runner
    windows-latest → Windows runner

-------------------------------------------------------------------------------------------------------
🔹 python-version: [3.8, 3.9, 3.10]

👉 Defines Python versions


💥 Total Jobs Created

Matrix expands like this:
OS  	    Python
Ubuntu	    3.8
Ubuntu	    3.9
Ubuntu	    3.10
Windows	    3.8
Windows	    3.9
Windows	    3.10
👉 Total = 6 parallel jobs

This is called:-> "Cartesian Product Execution"

-------------------------------------------------------------------------------------------------------

                                        🔶 steps:

👉 Sequence of actions inside each job
    Every matrix job runs all these steps independently
-------------------------------------------------------------------------------------------------------
                                    🔶 Step 1: Checkout Code
- name: Checkout code
  uses: actions/checkout@v3

👉 Pulls your repository into the runner

🧠 Simple Definition
In GitHub Actions, a runner is simply the machine that executes your workflow jobs.
-> A runner = execution environment
It:
    pulls your repo code
    installs dependencies
    runs your commands (npm test, pytest, etc.)
    reports results back to GitHub

What happens internally:
    GitHub clones your repo into VM

Why required:
    Without this:
    ❌ No code
    ❌ No tests
    ❌ No pipeline
-------------------------------------------------------------------------------------------------------
                                        🔶 Step 2: Setup Python

- name: Setup Python
  uses: actions/setup-python@v4

👉 Installs Python in the runner

🔸 with:
python-version: ${{ matrix.python-version }}

👉 Dynamically selects Python version

Example:
Each job gets:
                Job 1 → Python 3.8
                Job 2 → Python 3.9
                Job 3 → Python 3.10

Why critical:
    Different Python versions behave differently:
        Syntax support
        Dependency compatibility

👉 This ensures forward + backward compatibility
-------------------------------------------------------------------------------------------------------
                                🔶 Step 3: Cache pip Dependencies

- name: Cache pip
  uses: actions/cache@v3

👉 Speeds up pipeline execution

🔸 with:
        path: ~/.cache/pip
👉 Directory where pip stores downloaded packages

        key: ${{ runner.os }}-pip-${{ matrix.python-version }}
👉 Unique cache key

Example keys:
    ubuntu-pip-3.8
    windows-pip-3.10

Why key matters:
    Prevents mixing incompatible caches
    Ensures correct dependencies per environment

Real-world benefit:
Without cache:
    Every run downloads packages → slow ❌
With cache:
    Reuses previous installs → fast ✅
-------------------------------------------------------------------------------------------------------
                                    🔶 Step 4: Install Dependencies
run: |
  pip install -r requirements.txt

👉 Executes shell commands
    pip install -r requirements.txt
    Installs dependencies listed in file

Why:
    Your app depends on external libraries
    👉 CI must replicate production environment
-------------------------------------------------------------------------------------------------------
                                    🔶 Step 5: Lint Code

run: |
  cd python-app
  flake8 .

👉 Runs Flake8

What linting does:
    Checks code quality
    Finds:
        syntax errors
        bad practices
        style violations

Why it's important:
Prevents:
    ❌ messy code
    ❌ hidden bugs

👉 In companies:
Linting is a mandatory gate
-------------------------------------------------------------------------------------------------------
                                    🔶 Step 6: Run Tests

run: |
  cd python-app
  pytest

👉 Runs tests using pytest

What happens:
    Finds test files
    Executes test cases
    Reports pass/fail
Why critical:
    This is the final validation layer

👉 If tests fail:
        Pipeline fails
        Code is rejected
-------------------------------------------------------------------------------------------------------
                                    🔁 Full Execution Flow (Per Job)

For each of the 6 jobs:
    1. Spin up VM (Ubuntu/Windows)
    2. Install Python version
    3. Restore cache (if exists)
    4. Install dependencies
    5. Lint code
    6. Run tests
    7. Report result
-------------------------------------------------------------------------------------------------------
                                    🧠 Architect-Level Insights

1. This is Cross-Environment Testing
    Not just “run tests” → it's:
        Validate code across OS + runtime versions

2. Cache Key Design is Critical
    Bad keys = wrong dependencies = flaky builds

3. Fail-Fast Strategy is Intentional
        Debugging mode → false for debugging mode
        Fast pipelines → true for fast pipelines
4. Directory Context Matters
    Every cd python-app:
    Means repo is multi-project structured
-------------------------------------------------------------------------------------------------------

❌ Forgetting cache → slow pipelines
❌ Mixing OS-specific paths
❌ Not testing multiple Python versions
❌ No linting step
